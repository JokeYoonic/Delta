## 三、T2: 修复知识笔记收录逻辑

### 3.1 现状问题

当前 [AIChat.tsx](file:///Users/mac/Documents/trae_projects/Delta/app/src/sections/AIChat.tsx) 中的 BookmarkPlus 按钮逻辑存在以下问题：

| 问题                 | 严重度 | 说明                                                                                                                              |
| -------------------- | ------ | --------------------------------------------------------------------------------------------------------------------------------- |
| 无 Book 时静默失败   | 高     | `memoryBooksApi.list()` 返回空数组时，`book` 为 null，整个收录操作被跳过，用户点击无任何反馈                                  |
| 错误被吞掉           | 高     | `.catch(() => {})` 静默忽略所有错误，用户无法感知失败                                                                           |
| DOM 操作做提示       | 中     | 使用 `document.activeElement` + `setAttribute('title', ...)` 做轻提示，不可靠且不符合 React 范式                              |
| 查找问题消息逻辑有误 | 中     | `find((m, i) => i < currentConversation.messages.indexOf(msg) && m.role === 'user')` 只找第一条之前的 user 消息，应该是最近一条 |
| 类型不安全           | 低     | 多处 `any` 类型断言 `(book as any).id`                                                                                        |

### 3.2 任务拆解

#### T2.1 自动创建默认 Memory Book

- [ ] 后端 `memory_books.py`：新增 `POST /memory-books/ensure-default` 接口，如果用户没有 Book 则自动创建一个"我的知识笔记"
- [ ] 前端 `AIChat.tsx`：收录时先调用 `ensure-default`，保证一定有 Book 可用

#### T2.2 收录逻辑重构

- [ ] 将 BookmarkPlus 的 `onClick` 内联逻辑提取为 `handleBookmark(msg)` 函数
- [ ] 修复问题消息查找：改为从 `msg` 位置向前找最近一条 `role === 'user'` 的消息
- [ ] 错误处理：失败时展示 toast/系统消息，而非静默忽略
- [ ] 成功反馈：使用 store 中的状态或 toast 组件替代 DOM 操作

#### T2.3 类型安全

- [ ] `api/client.ts` 中 `memoryBooksApi` 的返回类型补充完整
- [ ] 移除 `any` 类型断言

### 3.3 验证标准

- [ ] 用户首次点击收录按钮，自动创建默认 Book 并成功收录
- [ ] 收录成功有明确视觉反馈（toast 或消息）

## 四、T3: 优化 Seed 脚本架构

### 4.1 现状问题

| 问题                       | 说明                                                                                                                                    |
| -------------------------- | --------------------------------------------------------------------------------------------------------------------------------------- |
| 知识硬编码在 Python 代码中 | `seed_knowledge.py` 和 `seed_full_subjects.py` 将全部知识点写在 `KNOWLEDGE_BASE` / `EXTRA` 字典中，更新需改代码                 |
| 两个脚本功能重叠           | `seed_knowledge.py` 包含数学/英语/语文，`seed_full_subjects.py` 包含历史/政治，但都调用同一个 `ragflow_service.upload_document()` |
| 无增量更新                 | 每次运行都是全量上传，无法判断哪些知识已存在                                                                                            |
| 无科目管理                 | 科目列表分散在两个脚本和前端 `AIChat.tsx` 的 `subjects` 数组中                                                                      |
| 爬取功能耦合               | `seed_knowledge.py` 中的 `crawl_url` 与知识填充是两个不同关注点                                                                     |

### 4.2 目标架构

```
backend/
  data/
    knowledge/                    # 知识点数据文件（与代码分离）
      数学.json                   # 每个科目一个文件
      英语.json
      语文.json
      历史.json
      政治.json
      物理.json
      化学.json
    subjects.json                 # 科目元数据（名称、图标、颜色、排序）
  scripts/
    seed_knowledge.py             # 统一入口脚本（合并两个脚本）
    crawl_knowledge.py            # 独立爬取脚本
  app/
    services/
      knowledge_service.py        # 知识库业务逻辑层
```

### 5.2 目标架构

采用 **Docker Compose Profile** 机制，按功能分类管理服务：

```yaml
# docker-compose.yml 结构
services:
  # ===== 核心基础设施（默认启动）=====
  postgres:     # profile: 无（默认）
  redis:        # profile: 无（默认）
  bifrost:      # profile: 无（默认）
  logto:        # profile: 无（默认）

  # ===== 语音服务（--profile voice）=====
  funasr:       # profile: [voice]
  kokoro:       # profile: [voice]
  faster-whisper: # profile: [voice]

  # ===== 应用服务（--profile app）=====
  backend:      # profile: [app]
  frontend:     # profile: [app]

  # ===== 全量启动（--profile full）=====
  # 包含以上所有
```

启动命令：

```bash
docker compose up -d                          # 只启动核心基础设施
docker compose --profile voice up -d          # 核心 + 语音服务
docker compose --profile app up -d            # 核心 + 应用服务
docker compose --profile full up -d           # 全部服务
```
